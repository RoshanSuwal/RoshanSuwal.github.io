Exception Response

26 July 2022

Exception Response is the response returned when error is encountered while serving the request

What does it contains?

  • status_code : it determines the type of error or exception triggered along with its severity

  • message : detail information about the error

  • ResponseMeta : information about the organization serving this system or service

Getting Started

16 May 2022

Here is how we register apis in apostle system.

Step 1

First step is to add base info that is applicable to all apis.Components that are involved are as follows.


1.1 Redirect To

It refers to the actual domain or IP of the api server. When the request is validated by apostle middleware system, it forwards the request to this value.


1.2 Base URL

This value is given to the user which they use to make the request. It differs from Redirect To is a sense that Redirect To is not known to the user, which makes the actual api server safe.Base URL uses a predefined format slug.apostle.ekbana.net where slug for project is set from Project Management System. The format is pre-configured so, it can be used as soon as the api is registered.However, the format cannot be changed. If someone needs to use other format, they should use Custom URL


1.3 Custom URL

Unlike BaseURL, custom URL is not pre-configured meaning it cannot be used right away. Server administration must be notified to make an apache conf file for the custom URL. Custom url is not restricted in any specific format so, you can use any format you need.


1.4 Protocol

Both HTTP and HTTPs protocol is supported. This protocol is used to forward the incoming request to api server.

Step 2

Now, we add additional information for the API.


2.1 Name

Unique identifier for the API.


2.2 Endpoint

Endpoint for the api. All value after the domain should be used as endpoint. If the part of endpoint is used in domain and the rest as endpoint, it will throw error during validation and might throw API not registered error.


2.3 Method

Currently, the system supports following methods :

  1. GET

  2. POST

  3. PUT

  4. DELETE

  5. HEAD

Choosing correct method is important. Incorrect method will result in API not registered error


2.4 Creator Email

If the validator system detect 5xx,4xx or any error message, this value will be used to send email to the creator


2.5 Is Cached

Defines if the response should be cached. Checking this value will give options for cache time interval as well as the parameters that will be used for creating cache key. Options include :

  1. Time
    Currently Minute, Hour and Day values are supported. Existing cache will be removed after this defined interval

  2. Cache headers
    If this value is set, the validator system will check in headers for this header-key and use the corresponding header-value to create cache key.

  3. Query parameters
    If this value is set, the validator system will check in URL for this query-key and use the corresponding query-value to create cache key.

If both above values are set, the validator system will use both to create cache key.


2.6 Request Add-ons

These values will be added to current request. The value that are set here will not be known to user only to validator system and api server. It is extremely important since it will prevent any attempt to bypass validator.

  1. Headers
    Values will be added as header parameters.

  2. Query Parameters
    Values will be added as query parameters.

Cache Response

16 May 2022

Cache Response is the response service that enables the caching of the response in order to decrease the response time of the request.

Why Caching Response ?

  • decrease the response time of request.

  • reduces the loads to backend servers.

How to configure Caching

How Caching is done ?

Only the responses from the cache enabled APIs are stored in cache.

Cacheable responses are stored using Hash data structure in Redis database.

  • Hash Key is created using the alias and api_id

  • Sub Hash Key is created using the caching headers param, caching query params and path variables specified while registering the API.

  • Response are set as values for the specific sub hash key for specific TTL period defined while registering API

Only the response with http status 200 are eligible for caching.

Publickey Validation

16 May 2022

Publickey Validation allows to send request from the valid client by validating Api-Key.

What is Api-Key?

Api-Key is a unique secret key (public key) generated when registering the group of Api (project) in the system.

How is Publickey Validated?

When client sends request, they should send the Api-Key in the header section. The provided Api_Key must match with the publickey registered in the system else it throws PublicKeyException.

PersonalPackage Validation

16 May 2022

PersonalPackage Validation validated the personal packaged. It is associated with the public key.

What is PersonalPackage?

PersonalPackage is the package defining the quota and lifespan of the public key. Quota includes the agreed total calls and grace calls allowed for specific api. And lifespan determines how long to serve the Api.

How and when is PersonalPackage validated?

PublicKey validated requests are set for personal package validation.

  • total calls usages and grace calls usages are stored in TOTAL_CALLS and GRACE_CALLS keyspace in redis database.

  • when the total call usages exceeds the agreed total calls, then the api is registered in GRACE_KEYSPACE and grace calls are started.

  • if used grace calls exceed the allowed grace calls, then public key is registered in EXPIRED_KEYSPACE.

  • when the current time exceed the lifespan of public key then the public key is expired and is registered in EXPIRED_KEYSPACE.

  • whenever a request is using public key which is in EXPIRED_KEYSPACE is sent, then it throws ExpiredKeyException.

IP Validation

16 May 2022

IP Validation validate the Ip address of the request. Only the requests from the valid IP source can proceed further

Classification of IP Addresses

1. Marked IP Address

Marked IP Address are the suspicious IP address that has performed some miscellaneous activities. These Ip address are closely monitored by the system and if further miscellaneous activities are performed, then it is blocked. These address can further send requests.

2. Blocked IP Address

When the suspicious/marked ip address further perform miscellaneous or illegal activities, then there address are marked as block ip address. They cannot further send requests to the system.

3. Valid IP Address

These are the Ip address from the valid sources. They are allowed to send requests to system.

How Validation is done?

All the blocked and marked Ip address are stores in BLOCKED_KEYSPACE AND MARKED_KEYSPACE in hash data structure in redis database. The system first checks if the given ip address is in BLOCKED_KEYSPACE, if present it returns the IPException else proceed forward.

A set of rules has been implemented which is responsible for analysing the ip address. It registers the IP address in MARKED_KEYSPACE and BLOCKED_KEYSPACE.

Funnel Validation

16 May 2022

Funnel Validation determine the rate of request to be served under given public key.

Why Funnel Validation?

  • determines the number of requests to be served in specific time frame.

How is Funnel Validation done?

  • A rate is defined when an api is registered under the group (public key).

  • supports request per minute, hour, day, month and year.

  • Whenever a request is served, the served request count is incremented by 1.

  • If the served request count exceed defined rate limit for given time interval, then it throws FunnelException.

How tracking of request rates is done?

  • the served request count is stored for specific interval of time in the redis database in following format

    {public_key_alias}{api-id}_{time-interval} : {served request count} TTL : {TTL duration}

  • when a request is successfully served then the call count in incremented by 1.

Api Validation

16 May 2022

Api Validation validates the api used in given request.

How is Api validated?

An Api consists of following parts:
. Base Url : www.xyz.com
. Request Method : GET, POST, PUT, DELETE, PATCH
. End Point : /abc/def

An Api is set to be valid Api if and only if Api with same base url, request method and endpoint is registered in the system else invalid.

ApiException is thrown by system when an invalid api request is received.

What is a group

16 May 2022

Groups are the collection of APIs that is used by the client.Api can belong to any projects. Thus, a group may contain APIs from multiple projects.Each group has its own public which must be used to send request to the validator.

Importance

  1. Groups can use package independently according to the users need.

  2. Grace amount can be set to different amount for each api in that group which won’t affect same api in different group

  3. Not restricted to same api. Different group can use same api from same projects.

  4. Separate usage dashboard.

  5. Public key is independent, so it can be blocked or assigned a new one for all apis in that group

Fields

Basic

  1. Group : Name of the group

  2. User email : This value will be used to send email to the User if following issues are detected :

    1. A week before public key expiration date.

    2. Allowed calls about to exhaust.

    3. Grace period started.

    4. Grace period ended and key is expired

    5. DDoS attempt

Selected APIs

API Name : same as defined during API registration.
Package : Here you can select base package for the API. Even in a group different API can have different Package.
Grace : Grace calls are the calls that are activated after paid calls are over. Paid calls are defined in a package. Grace calls are allocated so that user can get time to re-subscribe.

Public Key

Expires On : Expiration date of the group and associated public key.
Public Key : Public key is used by user while sending request to validator.

Apostle - overview

16 May 2022

Apostle is a reliable, secure and scalable way to publish, consume and manage API’s.It ensures optimal performance of the API’s, tracks and enforces usage, authentication, and more. This topic provides an overview of Apostle, including register your API, create group and monitor usage. For more information on specific topic please see relative link on sidebar.

In this topic we will walk you through the process of registering your first API and monitoring its usage.

Let’s do it step by step.

Step One : Register your api

  1. Click on project on the left sidebar.

  2. If you don’t see you project listed, you need to register it on Project Management System, and if you don’t have access to that system, please contact administrator for registration.

  3. Click on Add API.

For more info on the fields click here

Step two : Create a group

  1. Click on add group on client page

  2. Click on select API from project list. You can select api from multiple projects.

  3. Select the APIs and click on next.

  4. Enter group name and user email for this group.

  5. Select package and add grace calls if it applies.

  6. Generate public key and save

For more info on fields click here

Step three : Create package [ only necessary if you want to use custom pacakge ]

  1. Click on packages on left sidebar.

  2. Click on add package.

  3. Fill out necessary details.

For more info on fields click here.
